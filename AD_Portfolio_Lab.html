Step-by-Step Process
1
Open Server Manager Dashboard
Launched Server Manager on Windows Server 2019. Navigated to Add Roles and Features and selected the Active Directory Domain Services (AD DS) role to install.
2
Promote Server to Domain Controller
After AD DS installed, used the post-install wizard to promote the server to a Domain Controller. Created a new forest with the domain name jay.local. Set the DSRM (Directory Services Restore Mode) password and completed the promotion.
Domain: jay.local
3
Open Active Directory Users and Computers (ADUC)
Accessed ADUC via Tools → Active Directory Users and Computers in Server Manager. This is the main tool for managing all AD objects. Confirmed the domain structure was visible: jay.local with default containers.
Tools → Active Directory Users and Computers (dsa.msc)
4
Build Out the OU Structure
Right-clicked jay.local → New → Organisational Unit and created the OUs needed to organise the domain: Admins (delegated admin accounts), Accounts (general user accounts), IT and HR (departmental staff), IT_DEPARTMENT (domain-joined computer accounts), and Groups (security groups). Keeping these separate makes it possible to apply different policies and permissions to each.
New → Organisational Unit → Admins / Accounts / IT / HR / IT_DEPARTMENT / Groups
5
Create User Accounts
Navigated to the Accounts OU, right-clicked → New → User, and created several user accounts following a consistent naming convention. Set a temporary password for each and required a change at first login.
Example: JoE Martin → username: jmartin
6
Edit User Properties & Practise Enable/Disable
Opened a user's Properties dialog to edit details on the General and Organization tabs. Practised disabling an account from the Account tab — e.g. for a departing employee — then re-enabling it. Disabling blocks logon without deleting the account or its group memberships.
Right-click user → Properties / Disable Account
7
Create Departmental Security Groups
In the Groups OU, created seven security groups covering every department: Accounting, Sales, IT, Marketing, HR, Executive, and Finance. Each group controls access to that department's resources without managing permissions per individual user.
New → Group → Accounting / Sales / IT / Marketing / HR / Executive / Finance
8
Add a User to a Security Group
Opened the Finance group's Properties → Members tab → Add, and added JoE Martin. This grants Martin access to anything permissioned to the Finance group, without touching his account individually.
Finance Properties → Members → Add → JoE Martin
9
Remove a User from a Security Group
Practised the reverse operation: opened a group's Members tab, selected an existing member, and clicked Remove. Confirmed the user immediately lost the access tied to that group — the standard step when someone changes role or leaves a department.
Group Properties → Members → select user → Remove
10
Transfer a User from HR to IT
Modelled a department transfer end-to-end: moved a user object from the HR OU into the IT OU (right-click → Move), then updated group membership — removed from HR, added to IT. The OU move handles policy and location; the group change handles resource access — both are needed for a real transfer.
Move user: HR OU → IT OU, then update group membership
11
Reset a Password & Force Change at Next Logon
In ADUC, right-clicked a user and selected Reset Password to issue a new temporary password, ticking User must change password at next logon. This is the most common helpdesk request, and avoids the admin ever needing to know the user's next password.
Right-click user → Reset Password → ☑ User must change password at next logon
12
Review Account Lockouts & Login Failures
Worked through the Account tab to tell apart a locked-out account (too many failed attempts — fixed by ticking Unlock account) from other login failures, such as an expired password, a disabled account, or logon hours restrictions. Each looks the same to the user but needs a different fix.
Account tab → Unlock account / Account expires / Logon Hours
13
Set Up File Sharing with Share & NTFS Permissions
Created a shared folder on the server and shared it over the network. Set Share permissions at the share level, then set separate NTFS (Security tab) permissions on the folder itself, giving the relevant department's security group access while leaving others without it. Confirmed that when Share and NTFS permissions differ, the more restrictive one wins.
Properties → Sharing (Share) · Properties → Security (NTFS)
14
Delegate Administrative Control
Rather than relying on the built-in Administrator account for everything, created two named admin accounts — Jay Admin and Josh Mark — in the Admins OU. Used the Delegation of Control Wizard to grant one of them limited rights over a specific OU instead of full Domain Admin rights, in line with least-privilege practice.
Right-click OU → Delegate Control…
15
Join Computers to the Domain
On two client machines, went to System Properties → Computer Name → Change, switched from Workgroup to Domain, and joined jay.local using domain admin credentials. After rebooting, signed in with a domain user account on each to confirm authentication was working.
sysdm.cpl → Computer Name → Change → Domain: jay.local
16
Investigate a Trust Relationship Failure
Looked into the classic "trust relationship between this workstation and the domain has failed" error, which happens when a computer's AD account and its local record of that relationship fall out of sync — e.g. after a rebuild or a reset computer password. Checked basic connectivity first, then confirmed the two standard fixes: reset the computer account in ADUC, or remove and rejoin the domain.
ping <DC> · nslookup jay.local · ADUC → Reset Account
17
Create and Link a Group Policy Object
In Group Policy Management, created a new GPO and linked it to the HR OU. Under User Configuration → Administrative Templates → Control Panel, enabled Prohibit access to Control Panel and PC settings, then ran a policy update on a test client and checked the result to confirm it applied to HR users only.
gpupdate /force · gpresult /r

Commands Used
⌨️gpupdate /force— forces Group Policy to refresh immediately instead of waiting for the next cycle
⌨️gpresult /r— shows which GPOs are currently applied, to confirm a policy actually took effect
⌨️nslookup— queries DNS; the first check when a computer can't find the domain
⌨️ping— tests basic network connectivity to the domain controller or another host
⌨️hostname— shows the local computer's name, to confirm which machine you're on
⌨️whoami— shows the currently logged-in user (add /groups to see their memberships)

Key Concepts Practised
🏗️ Forest & Domain creation — A new AD forest is created when promoting the first DC. The forest root domain (jay.local) is the top of the entire AD hierarchy.
📁 Organisational Units (OUs) — OUs are containers that organise objects and allow different Group Policies to be applied to different sets of users and computers.
👥 Security Groups — Used to control resource access for a department. Instead of setting permissions per user, you set them on the group — all members inherit that access automatically.
🔐 Domain Authentication — When a user logs in on any domain-joined computer, the Domain Controller verifies their credentials from the AD database — not the local machine.
📋 Naming Conventions — Following a consistent naming convention for users and admin accounts is a sysadmin best practice that keeps a growing directory organised and predictable.
🔑 Group Membership — Adding or removing a user from a Security Group immediately grants or revokes whatever that group is permissioned to access — no need to touch the resource itself.
✏️ User Lifecycle Management — Creating an account is only the start. Editing properties, disabling a leaver's account, and moving someone's OU and group membership when they change departments are all part of keeping a directory accurate over time.
🔓 Password Support & Account Lockouts — Resetting a password and forcing a change at next logon covers most day-one helpdesk tickets. A lockout (too many bad attempts) and a login failure (expired or disabled account) look identical to the user but need different fixes.
🗄️ Share vs. NTFS (Security) Permissions — Share permissions control access over the network; NTFS permissions control access on disk. When a user hits both, the more restrictive of the two always wins.
🛡️ Delegation & Least Privilege — Rather than handing out the built-in Domain Admin account, delegating limited control over one OU to a named admin account keeps each admin's reach no bigger than the job requires.
📜 Group Policy Objects (GPOs) — A GPO linked to an OU applies a setting to every user or computer inside it at once — far faster than configuring a restriction machine by machine.
🖥️ Computer Accounts & Trust Relationships — Computers get their own AD account, just like users. When that account and the machine's local record fall out of sync, Windows throws a "trust relationship has failed" error — fixed by resetting the account or rejoining the domain.
🧭 Login Troubleshooting — Most domain login problems trace back to a short list of causes: expired or locked account, broken domain connectivity, or a computer account out of sync. Checking these in order narrows the problem fast.
